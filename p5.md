## 1. Purpose

Deliver three specialized adapters:
- **Scrape/RSS Adapter** — covers web scraping of semi-structured pages and RSS/Atom feed ingestion for sources without formal APIs. Targets: SIPRI (arms embargoes, publications), Janes, ICG CrisisWatch, HRW, Amnesty International, Genocide Watch, OPCW, Freedom House, Walk Free GSI, HURIDOCS, ICC Case Info. Tiers 14, 20, 21.
- **AIS/ADS-B Adapter** — covers real-time and near-real-time position tracking feeds for maritime (AIS) and aviation (ADS-B) traffic. Targets: ADS-B Exchange, OpenSky Network, AIS (MarineTraffic, MSSIS), FlightRadar24, FlightAware. Tier 18.
- **STIX/TAXII Adapter** — covers Structured Threat Information Expression (STIX 2.1) objects served via Trusted Automated eXchange of Indicator Information (TAXII 2.1) servers. Targets: MITRE ATT&CK, CISA KEV, AlienVault OTX, Abuse.ch, DISA STIGs. Tier 16.

All subclass `BaseAdapter` from P1 and produce `NormalizedRecord` output.

## 3. Scrape/RSS Adapter — `src/hydra/adapters/scrape_rss.py`

### 3.1 Class: `ScrapeRssAdapter(BaseAdapter)`

Handles two distinct ingestion patterns unified under a single adapter because they share the same operational profile: semi-structured content from sources that lack formal APIs. The adapter dispatches between scrape and RSS/Atom modes based on the registry `scrape_mode` field.

### 3.2 Ingestion Modes

| Mode | Mechanism | Use Case |
|---|---|---|
| `rss` | Fetch and parse RSS 2.0 or Atom 1.0 feeds. | SIPRI publications, ICG CrisisWatch, HRW reports, Amnesty International, OPCW news. |
| `atom` | Alias for `rss` — `feedparser` handles both transparently. | Sources declaring Atom feeds explicitly. |
| `scrape` | Fetch HTML pages and extract structured data via CSS selectors or XPath expressions declared in the registry. | Janes (defense intelligence), Genocide Watch alerts, Freedom House reports, SIPRI arms embargoes, Walk Free GSI, HURIDOCS, ICC Case Info. |

### 3.3 `fetch` Implementation

**RSS/Atom mode (`rss` or `atom`):**
- Reads `feed_url` from the stream registry entry.
- Fetches the feed via HTTP GET using `aiohttp.ClientSession`.
- Supports conditional fetch: if the registry declares `supports_conditional: true`, the adapter stores and sends `If-Modified-Since` and `If-None-Match` headers from the previous fetch. On HTTP 304, returns an empty `RawPayload` (content = empty bytes, http_status = 304). The `run` method short-circuits on 304.
- The `ETag` and `Last-Modified` values from the response are stored in the adapter's state for the next fetch cycle. Storage location: `{HydraSettings.data_dir}/feed_state/{stream_id}_feed_state.json`.
- Timeout from `HydraSettings.http_timeout_seconds` (default: 30).

**Scrape mode (`scrape`):**
- Reads `scrape_url` (or `scrape_urls` for multi-page sources) from the registry.
- Fetches each URL via HTTP GET.
- If the registry declares `scrape_follow_pagination: true`, the adapter follows pagination links. Pagination link discovery uses a CSS selector declared in `pagination_next_selector` (e.g., `a.next-page`, `li.pagination-next > a`). The adapter extracts the `href` attribute from the matched element, resolves it against the current page URL, and fetches the next page. Continues until no pagination element is found or `max_pages` (from registry, default: 10) is reached.
- If the registry declares `scrape_javascript_required: true`, the adapter logs a WARNING and falls back to static HTML fetch. JavaScript rendering is not supported at MVP — this field exists as a forward declaration for P11 (headless browser integration). Records from JS-required sources may be incomplete; the adapter tags them with `js_required: true` in the `RawPayload` headers for downstream awareness.
- Rate limiting: configurable `scrape_delay_seconds` (from registry, default: 2.0) between consecutive page fetches to the same domain. Enforced via `asyncio.sleep`.
- User-Agent: configurable per stream via registry `scrape_user_agent` field. Default: `HYDRA/{version} (+https://github.com/hydra-osint)`. Some sources block generic bot user agents — the registry can override with a browser-like UA string.

**Auth dispatch:**
- `none` — most RSS feeds and public pages.
- `api_key` — some sources require API key in header or query param (same mechanism as REST/JSON adapter).
- `basic_auth` — username/password from `HydraSettings.credentials[stream_id]`, injected as HTTP Basic Authentication header. Used for gated content behind simple auth walls.
- `cookie_auth` — the adapter reads a cookie name and value from `HydraSettings.credentials[stream_id]` and injects it as a `Cookie` header. Used for sources requiring session cookies obtained out-of-band.

### 3.4 `parse` Implementation

**RSS/Atom mode:**
- Parses the feed using `feedparser` library. `feedparser` handles RSS 2.0, RSS 1.0, Atom 1.0, and CDF transparently.
- Extracts feed-level metadata: `title`, `link`, `description`, `language`, `updated` (feed last-updated timestamp).
- Iterates `feed.entries`. Each entry becomes a dict with fields:
  - `title` — entry title.
  - `link` — entry permalink.
  - `published` — publication date (parsed by `feedparser` into a `time.struct_time`, converted to ISO 8601 UTC datetime).
  - `updated` — last-updated date if different from published.
  - `summary` — entry summary/description (HTML content). Stripped to plain text via `html.parser` if registry declares `strip_html: true` (default: true).
  - `content` — full entry content if present (some feeds include full article text).
  - `author` — entry author.
  - `categories` — list of category/tag strings.
  - `enclosures` — list of attached media (URL, type, length) if present.
  - `guid` — globally unique identifier for the entry (RSS `guid` or Atom `id`).
- If the registry declares `rss_content_fetch: true`, the adapter follows each entry's `link` URL to fetch the full article HTML page, then applies the scrape extraction rules (below) to extract structured content from the article page. This bridges RSS metadata with full-text scrape extraction.

**Scrape mode:**
- Parses the HTML response using `beautifulsoup4` with the `lxml` parser backend.
- Extraction is entirely configuration-driven via registry fields:
  - `scrape_item_selector` — CSS selector identifying the repeating content items on the page (e.g., `div.article-card`, `tr.data-row`, `li.report-item`). Each matched element becomes one record.
  - `scrape_field_map` — dict mapping output field names to extraction rules. Each rule is a dict with:
    - `selector` — CSS selector relative to the item element (e.g., `h3.title`, `span.date`, `a.link`).
    - `attribute` — which attribute to extract. Options: `text` (inner text, default), `href`, `src`, `data-*`, or any HTML attribute name.
    - `regex` — optional regex applied to the extracted string for further refinement (e.g., extracting a date from a mixed-content string).
    - `transform` — optional transformation: `strip`, `lowercase`, `uppercase`, `parse_date` (attempts ISO 8601, then common date formats via `dateutil.parser`), `parse_int`, `parse_float`.
  - `scrape_base_url_resolve` — if `true` (default: true), relative URLs extracted from `href` or `src` attributes are resolved against the page URL using `urllib.parse.urljoin`.
- If no items match `scrape_item_selector`, the adapter logs at WARNING and returns an empty list (the page structure may have changed — this is expected for scrape targets).

**Common post-parse step:**
- Each parsed record is tagged with `source_url` (the feed URL or page URL) and `fetch_mode` (`rss`, `atom`, or `scrape`) for provenance.

### 3.5 `validate` Implementation

- Required-field validation per registry `required_fields`.
- Date validation: if a `published` or `updated` field is present, verify it parses to a valid datetime. Records with unparseable dates are logged at WARNING; the field is set to `None` rather than dropping the record (dates from scrape targets are frequently malformed).
- URL validation: if `link` or extracted URL fields are present, verify they are well-formed URLs (scheme + netloc minimum). Malformed URLs are logged and set to `None`.
- Content deduplication: uses `guid` (for RSS/Atom) as the primary dedup key. For scrape mode, uses `compute_raw_hash` on the concatenation of all extracted field values. Intra-batch duplicates are dropped.
- Staleness check: if the registry declares `max_age_hours` (default: none), records with `published` timestamps older than `now - max_age_hours` are dropped. This prevents re-ingesting ancient feed entries on first fetch.

### 3.6 Stream Registry Entry Schema (Scrape/RSS-specific fields)

| Field | Type | Required | Description |
|---|---|---|---|
| `scrape_mode` | `enum[rss, atom, scrape]` | yes | Ingestion mode. |
| `feed_url` | `str` | no | RSS/Atom feed URL. Required if `scrape_mode: rss` or `atom`. |
| `scrape_url` | `str` | no | Single page URL to scrape. Required if `scrape_mode: scrape` and `scrape_urls` not set. |
| `scrape_urls` | `list[str]` | no | Multiple page URLs to scrape. For multi-page sources. |
| `scrape_item_selector` | `str` | no | CSS selector for repeating content items. Required if `scrape_mode: scrape`. |
| `scrape_field_map` | `dict[str, dict]` | no | Field extraction rules. Required if `scrape_mode: scrape`. |
| `scrape_base_url_resolve` | `bool` | no | Resolve relative URLs against page URL. Default: true. |
| `scrape_follow_pagination` | `bool` | no | Follow pagination links. Default: false. |
| `pagination_next_selector` | `str` | no | CSS selector for the "next page" link. Required if `scrape_follow_pagination: true`. |
| `scrape_delay_seconds` | `float` | no | Delay between consecutive page fetches. Default: 2.0. |
| `scrape_user_agent` | `str` | no | Custom User-Agent header for scrape requests. |
| `scrape_javascript_required` | `bool` | no | Forward declaration — source requires JS rendering. Default: false. |
| `strip_html` | `bool` | no | Strip HTML tags from text content. Default: true. |
| `rss_content_fetch` | `bool` | no | Follow entry links to fetch full article content. Default: false. |
| `max_age_hours` | `int` | no | Drop entries older than this many hours. Default: none (no staleness filter). |
| `max_pages` | `int` | no | Maximum pages to follow during pagination. Default: 10. |
| `auth_pattern` | `enum[none, api_key, basic_auth, cookie_auth]` | yes | Authentication strategy. |

## 4. AIS/ADS-B Adapter — `src/hydra/adapters/ais_adsb.py`

### 4.1 Class: `AisAdsbAdapter(BaseAdapter)`

Handles real-time and near-real-time position tracking data for maritime vessels (Automatic Identification System — AIS) and aircraft (Automatic Dependent Surveillance–Broadcast — ADS-B). These are high-frequency, high-volume streams with sub-minute cadence producing position reports, identity information, and kinematic data.

### 4.2 Ingestion Modes

Three modes, selected via registry `tracking_mode` field:

| Mode | Mechanism | Use Case |
|---|---|---|
| `rest_poll` | Periodic HTTP GET to a REST API returning JSON position reports. | ADS-B Exchange API, OpenSky Network REST API, FlightRadar24 API, FlightAware AeroAPI, MarineTraffic API. |
| `websocket` | Persistent WebSocket connection receiving streaming JSON messages. | ADS-B Exchange WebSocket feed, OpenSky Network live feed. |
| `raw_nmea` | Fetch or receive raw NMEA 0183 sentences (AIS) for decoding. | MSSIS (Maritime Safety and Security Information System), direct AIS receiver feeds. |

### 4.3 `fetch` Implementation

**REST poll mode (`rest_poll`):**
- Reads `base_url`, `endpoint_path`, and `query_params` from the stream registry entry.
- Constructs the request URL identically to the REST/JSON adapter (P1).
- Supports geographic bounding box filtering: registry fields `bbox_lat_min`, `bbox_lat_max`, `bbox_lon_min`, `bbox_lon_max` are injected as query parameters using the parameter names declared in `bbox_param_map` (e.g., `{"lat_min": "lamin", "lat_max": "lamax", "lon_min": "lomin", "lon_max": "lomax"}` for OpenSky).
- Supports aircraft/vessel filtering: registry field `icao_filter` (list of ICAO 24-bit addresses for ADS-B) or `mmsi_filter` (list of MMSI numbers for AIS) injected as query parameters.
- Response is JSON — an array of position reports or a wrapper object containing an array.
- Pagination: most tracking APIs return all current positions in a single response (no pagination). If the registry declares `pagination_type`, follows the same pagination logic as REST/JSON adapter.

**WebSocket mode (`websocket`):**
- Reads `websocket_url` from the registry.
- Opens a persistent WebSocket connection using `aiohttp.ClientSession.ws_connect`.
- Receives messages in a loop. Each message is a JSON object representing one position update.
- The adapter accumulates messages for a configurable `ws_batch_duration_seconds` (from registry, default: 10) before returning a single `RawPayload` containing all accumulated messages as a JSON array.
- Heartbeat: sends a ping every `ws_ping_interval_seconds` (from registry, default: 30) to keep the connection alive.
- Reconnection: on connection drop, the adapter waits `ws_reconnect_delay_seconds` (from registry, default: 5) and reconnects. Maximum reconnection attempts: `ws_max_reconnects` (from registry, default: 10). After exhausting reconnects, raises `FetchError`.
- Connection lifecycle: the WebSocket connection is opened on the first `fetch` call and reused across subsequent calls within the same adapter instance. The connection is closed when the adapter is garbage-collected or explicitly shut down.
- Subscription messages: some WebSocket feeds require an initial subscription message after connection. If the registry declares `ws_subscribe_message` (a JSON object), the adapter sends it immediately after connection establishment.

**Raw NMEA mode (`raw_nmea`):**
- AIS data is transmitted as NMEA 0183 sentences — ASCII text lines beginning with `!AIVDM` or `!AIVDO`.
- Two sub-modes based on registry `nmea_source_type`:
  - `http_stream` — connects to an HTTP endpoint that streams NMEA sentences (one per line). Uses chunked transfer encoding. The adapter reads lines for `nmea_batch_duration_seconds` (from registry, default: 10) before returning a `RawPayload`.
  - `file_fetch` — fetches a file containing accumulated NMEA sentences (e.g., from MSSIS bulk download). Standard HTTP GET.
- Each NMEA sentence is a self-contained message, but multi-fragment messages (message types 5, 24) span multiple sentences linked by a sequential message ID and fragment number. The adapter reassembles multi-fragment messages before parsing.

**Auth dispatch:**
- `none` — OpenSky Network (limited), some AIS feeds.
- `api_key` — ADS-B Exchange, FlightRadar24, FlightAware AeroAPI, MarineTraffic. Key injected per registry `auth_key_location` and `auth_key_name`.
- `basic_auth` — MSSIS and some institutional AIS feeds.
- `rapidapi_key` — ADS-B Exchange via RapidAPI. Injects `X-RapidAPI-Key` and `X-RapidAPI-Host` headers from `HydraSettings.credentials[stream_id]`.

### 4.4 `parse` Implementation

**JSON position reports (REST poll and WebSocket modes):**
- Decodes JSON response.
- Extracts the position array from the response using `response_root_path` from registry (e.g., `states` for OpenSky, `ac` for ADS-B Exchange, `data` for MarineTraffic).
- Each position report becomes a dict. Field names vary by source — the adapter applies a `field_mapping` from the registry to normalize source-specific field names to a canonical set.

**Canonical ADS-B fields (post-mapping):**

| Canonical Field | Type | Description |
|---|---|---|
| `icao24` | `str` | ICAO 24-bit transponder address (hex). |
| `callsign` | `str` | Aircraft callsign (flight number). |
| `latitude` | `float` | WGS84 latitude in degrees. |
| `longitude` | `float` | WGS84 longitude in degrees. |
| `altitude_m` | `float` | Barometric or geometric altitude in meters. |
| `velocity_ms` | `float` | Ground speed in meters/second. |
| `heading` | `float` | True track angle in degrees (0 = north, clockwise). |
| `vertical_rate_ms` | `float` | Vertical rate in meters/second (positive = climbing). |
| `on_ground` | `bool` | Whether the aircraft is on the ground. |
| `squawk` | `str` | Transponder squawk code (4-digit octal). |
| `timestamp` | `datetime` | Position report timestamp (UTC). |
| `source` | `str` | Data source identifier. |

**Canonical AIS fields (post-mapping):**

| Canonical Field | Type | Description |
|---|---|---|
| `mmsi` | `str` | Maritime Mobile Service Identity (9-digit). |
| `imo` | `str` | IMO ship identification number. |
| `vessel_name` | `str` | Vessel name. |
| `ship_type` | `int` | AIS ship type code (0–99). |
| `latitude` | `float` | WGS84 latitude in degrees. |
| `longitude` | `float` | WGS84 longitude in degrees. |
| `speed_knots` | `float` | Speed over ground in knots. |
| `course` | `float` | Course over ground in degrees. |
| `heading` | `float` | True heading in degrees. |
| `nav_status` | `int` | Navigational status code (0–15). |
| `destination` | `str` | Reported destination. |
| `eta` | `datetime` | Estimated time of arrival. |
| `draught` | `float` | Current draught in meters. |
| `timestamp` | `datetime` | Position report timestamp (UTC). |
| `source` | `str` | Data source identifier. |

**Raw NMEA parsing:**
- Decodes NMEA sentences using `pyais` library.
- NMEA sentence structure: `!AIVDM,{fragment_count},{fragment_number},{sequential_id},{channel},{payload},{fill_bits}*{checksum}`.
- The adapter:
  1. Validates the NMEA checksum (XOR of all characters between `!` and `*`).
  2. Groups multi-fragment messages by `sequential_id`.
  3. Reassembles the payload from all fragments.
  4. Decodes the 6-bit ASCII-armored payload into binary.
  5. Parses the binary payload according to the AIS message type.
- Supported AIS message types:

| Type | Name | Fields Extracted |
|---|---|---|
| 1, 2, 3 | Position Report (Class A) | mmsi, nav_status, speed_knots, longitude, latitude, course, heading, timestamp. |
| 5 | Static and Voyage Data | mmsi, imo, callsign, vessel_name, ship_type, draught, destination, eta. |
| 18 | Position Report (Class B) | mmsi, speed_knots, longitude, latitude, course, heading, timestamp. |
| 24 | Class B Static Data | mmsi, vessel_name, ship_type, callsign. |

- Message types not in the supported list are logged at DEBUG and skipped.
- Position reports (types 1, 2, 3, 18) and static data (types 5, 24) are parsed into separate dicts. The adapter does NOT attempt to merge position and static data at parse time — that is a correlation engine concern (P9). Each message produces one dict.

**Common post-parse step:**
- Each parsed record is tagged with `tracking_mode` (`rest_poll`, `websocket`, or `raw_nmea`), `data_domain` (`adsb` or `ais`), and `source_api` (the specific API name from registry) for provenance.

### 4.5 `validate` Implementation

- Coordinate validation: `latitude` in [-90, 90], `longitude` in [-180, 180]. Records with coordinates at exactly (0, 0) are flagged — this is a common AIS default/error position. If registry declares `reject_null_island: true` (default: true), records at (0.0, 0.0) are dropped.
- Speed validation: `speed_knots` or `velocity_ms` must be non-negative. Upper bound from registry `max_speed` (default: 600 m/s for ADS-B, 50 knots for AIS). Records exceeding max speed are dropped (likely erroneous).
- Altitude validation (ADS-B only): `altitude_m` in range [-500, 100000] (below sea level for Dead Sea area airports, up to ~100km for high-altitude balloons). Out-of-range values dropped.
- MMSI validation (AIS only): must be a 9-digit string matching known MMSI format patterns. MID (Maritime Identification Digits) in positions 1-3 must be in the valid range (201–775). Invalid MMSIs logged at WARNING and dropped.
- ICAO24 validation (ADS-B only): must be a 6-character hexadecimal string. Invalid values dropped.
- Timestamp validation: must be within `max_position_age_seconds` (from registry, default: 300 for ADS-B, 3600 for AIS) of the current time. Stale positions are dropped — they represent cached/replayed data, not live tracks.
- Deduplication: composite key of `{icao24|mmsi}_{timestamp}` (rounded to nearest second). Duplicate position reports within the same second are dropped.

## 5. STIX/TAXII Adapter — `src/hydra/adapters/stix_taxii.py`

### 5.1 Class: `StixTaxiiAdapter(BaseAdapter)`

Handles ingestion of cyber threat intelligence data encoded as STIX 2.1 (Structured Threat Information Expression) objects, served via TAXII 2.1 (Trusted Automated eXchange of Indicator Information) servers or direct STIX bundle downloads. STIX/TAXII is the dominant standard for machine-readable CTI sharing.

### 5.2 TAXII 2.1 Protocol Overview

TAXII 2.1 defines a RESTful API for CTI exchange:

| Endpoint | Path | Returns | Purpose |
|---|---|---|---|
| Discovery | `{taxii_url}/taxii2/` | Server info, API roots | Enumerate available API roots. |
| API Root | `{api_root}/` | API root info, title, versions | Metadata about the API root. |
| Collections | `{api_root}/collections/` | List of collections | Enumerate available STIX collections. |
| Collection | `{api_root}/collections/{id}/` | Collection metadata | Title, description, media types. |
| Objects | `{api_root}/collections/{id}/objects/` | STIX bundle (envelope) | Retrieve STIX objects from a collection. |
| Object by ID | `{api_root}/collections/{id}/objects/{object_id}/` | Single STIX object | Retrieve a specific object. |
| Manifest | `{api_root}/collections/{id}/manifest/` | Object metadata list | List object IDs, types, versions without full content. |

### 5.3 `fetch` Implementation

**TAXII 2.1 mode (`taxii`):**
- Uses the `taxii2-client` library (already declared in P0 `pyproject.toml`).
- Reads `taxii_url`, `api_root_path`, and `collection_id` from the stream registry entry.
- Discovery flow (executed once per adapter instance, cached):
  1. If `api_root_path` is not declared, fetches the discovery endpoint (`{taxii_url}/taxii2/`) to enumerate API roots. Uses the first API root or the one matching `api_root_title` from registry.
  2. Fetches the collections list from `{api_root}/collections/`.
  3. If `collection_id` is not declared, uses the first collection or the one matching `collection_title` from registry.
- Object retrieval:
  - Fetches objects from `{api_root}/collections/{collection_id}/objects/`.
  - Supports filtering via TAXII query parameters:
    - `added_after` — ISO 8601 timestamp. Maps to `{last_fetch_time}` for incremental ingestion.
    - `match[type]` — filter by STIX object type (e.g., `indicator`, `malware`, `attack-pattern`). From registry `stix_type_filter`.
    - `match[id]` — filter by specific STIX object ID. From registry `stix_id_filter`.
    - `match[spec_version]` — filter by STIX version. Default: `2.1`.
  - Response is a TAXII envelope: `{"more": bool, "next": str, "objects": [...]}`.
  - Pagination: if `more` is `true`, the adapter follows `next` as a query parameter on the next request. Continues until `more` is `false` or `max_pages` (from registry, default: 100) is reached.
  - Content-Type: `application/taxii+json;version=2.1`.

**Direct STIX bundle mode (`stix_bundle`):**
- For sources that publish STIX bundles as static files or via non-TAXII REST endpoints (e.g., MITRE ATT&CK STIX repository on GitHub, CISA KEV as STIX bundle, Abuse.ch threat feeds converted to STIX).
- Reads `bundle_url` from the stream registry entry.
- Fetches the bundle via HTTP GET. Response is a JSON object conforming to the STIX 2.1 Bundle schema: `{"type": "bundle", "id": "bundle--{uuid}", "objects": [...]}`.
- Supports conditional fetch via `If-Modified-Since` / `If-None-Match` headers (same mechanism as Scrape/RSS adapter §3.3). State stored at `{HydraSettings.data_dir}/stix_state/{stream_id}_bundle_state.json`.
- For large bundles (MITRE ATT&CK is ~15MB), the adapter streams the response and parses incrementally using `ijson` library to avoid loading the entire bundle into memory at once.
- If the registry declares `bundle_format: stix_csv`, the adapter fetches a CSV file (e.g., CISA KEV CSV export) and converts each row to a STIX Indicator or Vulnerability object using a `csv_to_stix_mapping` declared in the registry. This is a convenience mode for sources that publish structured CTI data in non-STIX formats.

**Auth dispatch:**
- `none` — MITRE ATT&CK GitHub raw files, public STIX feeds.
- `api_key` — AlienVault OTX (OTX API key in `X-OTX-API-KEY` header), Abuse.ch.
- `basic_auth` — some institutional TAXII servers.
- `certificate` — TAXII servers requiring mutual TLS. The adapter reads `cert_path` and `key_path` from `HydraSettings.credentials[stream_id]` and passes them to the HTTP client's SSL context. Certificate files must be PEM-encoded.

### 5.4 `parse` Implementation

**STIX 2.1 object parsing:**
- The adapter processes STIX objects from either TAXII envelopes or direct bundles uniformly.
- Each STIX object is a JSON dict with a mandatory `type` field. The adapter handles the following STIX Domain Objects (SDOs) and STIX Cyber-observable Objects (SCOs):

| STIX Type | Category | Fields Extracted |
|---|---|---|
| `attack-pattern` | SDO | id, name, description, kill_chain_phases, external_references (MITRE ATT&CK ID), created, modified. |
| `campaign` | SDO | id, name, description, aliases, first_seen, last_seen, created, modified. |
| `indicator` | SDO | id, name, description, pattern (STIX pattern string), pattern_type, valid_from, valid_until, kill_chain_phases, created, modified. |
| `malware` | SDO | id, name, description, malware_types, is_family, aliases, first_seen, last_seen, created, modified. |
| `threat-actor` | SDO | id, name, description, threat_actor_types, aliases, roles, goals, sophistication, resource_level, primary_motivation, created, modified. |
| `vulnerability` | SDO | id, name, description, external_references (CVE ID), created, modified. |
| `tool` | SDO | id, name, description, tool_types, aliases, kill_chain_phases, created, modified. |
| `intrusion-set` | SDO | id, name, description, aliases, first_seen, last_seen, goals, resource_level, primary_motivation, created, modified. |
| `relationship` | SRO | id, relationship_type, source_ref, target_ref, description, created, modified. |
| `sighting` | SRO | id, sighting_of_ref, observed_data_refs, first_seen, last_seen, count, created, modified. |
| `identity` | SDO | id, name, identity_class, sectors, contact_information, created, modified. |
| `ipv4-addr` | SCO | id, value. |
| `ipv6-addr` | SCO | id, value. |
| `domain-name` | SCO | id, value. |
| `url` | SCO | id, value. |
| `file` | SCO | id, name, hashes (MD5, SHA-1, SHA-256), size. |
| `email-addr` | SCO | id, value, display_name. |

- STIX types not in the above list are parsed generically: the adapter extracts `id`, `type`, `created`, `modified`, and stores the remaining fields as-is in the payload.
- `external_references` are flattened: each reference with a `source_name` of `mitre-attack` has its `external_id` extracted as `mitre_attack_id`. References with `source_name` of `cve` have their `external_id` extracted as `cve_id`.
- `kill_chain_phases` are flattened to a list of strings in format `{kill_chain_name}:{phase_name}`.
- STIX pattern strings (in `indicator` objects) are stored as-is — pattern parsing is a correlation engine concern (P9).

**Common post-parse step:**
- Each parsed record is tagged with `stix_version` (`2.1`), `fetch_mode` (`taxii` or `stix_bundle`), and `taxii_collection_id` (if applicable) for provenance.

### 5.5 `validate` Implementation

- STIX ID format validation: must match `{type}--{uuid}` where `{type}` matches the object's `type` field and `{uuid}` is a valid UUID v4 or v5. Invalid IDs are logged at WARNING and the record is dropped.
- Required fields per type: `id`, `type`, `created`, `modified` are mandatory for all SDOs/SROs. SCOs require `id`, `type`, and at least one identifying property (`value`, `name`, or `hashes`). Records missing required fields are dropped.
- Timestamp validation: `created` and `modified` must be valid ISO 8601 timestamps. `modified` must be ≥ `created`. Records violating this are logged at WARNING; `modified` is set to `created` rather than dropping the record.
- STIX pattern validation (indicators only): if the registry declares `validate_stix_patterns: true` (default: false — pattern validation is expensive), the adapter validates the STIX pattern string using the `stix2-patterns` library. Invalid patterns are logged at WARNING; the record is retained but tagged with `pattern_valid: false`.
- Relationship validation: `source_ref` and `target_ref` must be valid STIX IDs. The adapter does NOT validate that the referenced objects exist in the current batch — cross-reference validation is a correlation engine concern (P9).
- Deduplication: uses the STIX `id` as the primary dedup key. If multiple versions of the same object exist (same `id`, different `modified` timestamps), only the most recent version is retained.
- Revocation handling: STIX 2.1 supports object revocation via the `revoked` boolean field. If an object has `revoked: true`, the adapter retains it but tags it with `revoked: true` in the output. Downstream consumers decide whether to act on revoked objects.

## 6. NormalizedRecord Mapping

All three adapters produce `NormalizedRecord` output via the `normalize` method inherited from `BaseAdapter`. The mapping from adapter-specific parsed dicts to `NormalizedRecord` fields:

### 6.1 Scrape/RSS → NormalizedRecord

| NormalizedRecord Field | Source |
|---|---|
| `stream_id` | From stream registry. |
| `tier` | From stream registry. |
| `timestamp` | `published` or `updated` datetime. Falls back to fetch time if neither available. |
| `geo` | `None` — scrape/RSS sources rarely carry geospatial data. If the registry declares `geo_extract_mode: regex`, the adapter attempts to extract coordinates from text content using regex patterns. |
| `payload` | Full parsed dict (all extracted fields). |
| `source_url` | Feed URL or page URL. |
| `source_name` | From stream registry `source_name`. |
| `raw_hash` | `compute_raw_hash` on `guid` (RSS/Atom) or concatenated field values (scrape). |
| `confidence` | `0.6` for scrape mode (HTML structure may change), `0.8` for RSS/Atom mode (structured feed format). Overridable via registry `default_confidence`. |
| `tags` | `[scrape_mode, source_name]` + registry `default_tags` + extracted `categories` (RSS/Atom). |

### 6.2 AIS/ADS-B → NormalizedRecord

| NormalizedRecord Field | Source |
|---|---|
| `stream_id` | From stream registry. |
| `tier` | From stream registry (Tier 18). |
| `timestamp` | Position report timestamp (UTC). |
| `geo` | `{"type": "Point", "coordinates": [longitude, latitude]}` — GeoJSON Point. For ADS-B, altitude is included as a third coordinate: `[longitude, latitude, altitude_m]`. |
| `payload` | Full canonical field dict (all ADS-B or AIS fields). |
| `source_url` | API endpoint URL. |
| `source_name` | From stream registry `source_name`. |
| `raw_hash` | `compute_raw_hash` on `{icao24|mmsi}_{timestamp_iso}`. |
| `confidence` | `0.9` for REST API sources, `0.85` for WebSocket (potential message loss), `0.7` for raw NMEA (decoding errors possible). Overridable via registry `default_confidence`. |
| `tags` | `[data_domain, tracking_mode, source_api]` + registry `default_tags`. |

### 6.3 STIX/TAXII → NormalizedRecord

| NormalizedRecord Field | Source |
|---|---|
| `stream_id` | From stream registry. |
| `tier` | From stream registry (Tier 16). |
| `timestamp` | STIX `modified` timestamp (most recent version). Falls back to `created`. |
| `geo` | `None` — STIX objects are not inherently geospatial. If the object contains location references (e.g., `identity` with `contact_information`), extraction is a correlation engine concern (P9). |
| `payload` | Full parsed STIX object dict. |
| `source_url` | TAXII collection URL or bundle URL. |
| `source_name` | From stream registry `source_name`. |
| `raw_hash` | `compute_raw_hash` on STIX `id` + `modified` timestamp. |
| `confidence` | `0.95` for TAXII (authoritative CTI sources), `0.85` for direct bundles (may be stale). Overridable via registry `default_confidence`. |
| `tags` | `[stix_version, stix_type, fetch_mode]` + `mitre_attack_id` or `cve_id` if present + registry `default_tags`. |

## 12. Notes & Forward References

- **JavaScript rendering** (scrape targets requiring JS): declared as `scrape_javascript_required` field for forward compatibility. Headless browser integration planned for P11. MVP uses static HTML fetch with degradation warning.
- **AIS position/static data correlation**: the AIS/ADS-B adapter produces separate records for position reports and static data messages. Merging vessel identity with position tracks is a P9 (Correlation Engine) responsibility.
- **STIX pattern parsing**: stored as raw strings. Pattern decomposition into observables for matching against live data is a P9 responsibility.
- **STIX relationship graph construction**: `relationship` and `sighting` objects reference other STIX objects by ID. Building the threat intelligence graph is a P9 responsibility.
- **Rate limiting for tracking APIs**: ADS-B Exchange and FlightRadar24 enforce strict rate limits. The BaseAdapter retry policy (P1) handles 429 responses with `Retry-After` header. Stream cadence in the registry should be set to respect documented API rate limits.
- **WebSocket lifecycle management**: the adapter opens the WebSocket on first `fetch` and reuses it. The scheduler (P8) must be aware that WebSocket-mode streams are long-lived and should not be treated as discrete fetch-sleep-fetch cycles. P8 will need a `streaming` cadence mode.
