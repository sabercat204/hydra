## 1. Purpose

Build the observability layer that makes HYDRA's operational state visible, alertable, and forecastable. P12 owns:

1. **Metrics exposition** — Prometheus-compatible `/metrics` endpoint on the FastAPI application, plus background collectors that scrape internal state into Prometheus gauges, counters, and histograms.
2. **Alert rule definitions** — Prometheus Alertmanager rules for backpressure thresholds, dead streams, SLA misses, job failures, DLQ depth, storage health degradation, and API error rates.
3. **Dashboard definitions** — Grafana JSON dashboard models for operational overview, adapter health, storage health, API performance, and intelligence product metrics.
4. **SLO/SLI definitions** — formal service level objectives for adapter health, product generation, and API availability with error budget tracking.
5. **Anomaly detection** — statistical anomaly detection on correlation volume and confidence drift using z-score and EWMA methods.
6. **Capacity planning** — storage growth projection, query latency trend analysis, and ingestion throughput forecasting.

After P12, HYDRA is fully observable — operators receive alerts before failures cascade, dashboards surface operational state at a glance, and capacity planning prevents resource exhaustion.

## 2. Design Decisions

### 2.1 Prometheus Client Library — Both

Use `prometheus_client` for custom application metrics (counters, gauges, histograms defined in code) and `prometheus_fastapi_instrumentator` for automatic HTTP request instrumentation on the P11 FastAPI application. The instrumentator provides request duration histograms, request counts by status code, and in-progress request gauges with zero code changes to routers.

### 2.2 Alert Manager — Standalone Alertmanager

Self-hosted Prometheus Alertmanager in Docker Compose. Cloud-native alternatives (CloudWatch Alarms, SNS) introduce AWS coupling that conflicts with HYDRA's infrastructure-agnostic design. Alertmanager provides grouping, inhibition, silencing, and multi-receiver routing natively.

### 2.3 Dashboard Tooling — Grafana (Self-Hosted)

Grafana OSS in Docker Compose with provisioned datasources and dashboards. Dashboards are version-controlled JSON files, not manually created. CloudWatch Dashboards would require AWS coupling and cannot query Prometheus directly.

### 2.4 Metric Retention Policy

- **Raw resolution**: 15-second scrape interval, retained 15 days in Prometheus.
- **Downsampled long-term**: recording rules aggregate to 5-minute and 1-hour resolution. 5-minute retained 90 days, 1-hour retained 1 year.
- **Prometheus storage**: local TSDB with `--storage.tsdb.retention.time=15d`. Long-term storage (Thanos/Cortex) deferred to post-MVP.

### 2.5 Metric Naming Convention

All custom metrics use `hydra_` prefix with subsystem labels:

```
hydra_{subsystem}_{metric_name}_{unit}
```

Subsystems: `adapter`, `storage`, `scheduler`, `api`, `correlation`, `product`, `dlq`, `backpressure`.

Labels follow Prometheus best practices: low cardinality, no unbounded values. Standard labels: `tier`, `cadence`, `engine`, `stream_id` (capped to active streams only), `pipeline_id`, `product_type`, `endpoint`, `method`, `status_code`.

### 2.6 Log Aggregation — Structured JSON to stdout

12-factor compliant: all application logs emit structured JSON to stdout. Docker/Kubernetes log drivers collect and forward. No dedicated log pipeline for MVP. Log fields: `timestamp`, `level`, `module`, `message`, `stream_id`, `tier`, `engine`, `request_id`, `duration_ms`, `error`.

Structured logging configuration added to `HydraSettings.monitoring`.

### 4.3 Collector Architecture

Background collectors run as `asyncio.Task` instances started at app startup. Each collector scrapes a specific subsystem on a configurable interval and updates Prometheus metrics in the `prometheus_client` registry.

```python
class BaseCollector(ABC):
    """Abstract base for metric collectors."""

    def __init__(self, interval: float = 60.0):
        self._interval = interval
        self._running = False

    async def start(self) -> asyncio.Task:
        """Start the collection loop as a background task."""
        self._running = True
        return asyncio.create_task(self._loop())

    async def stop(self) -> None:
        """Signal the collection loop to stop."""
        self._running = False

    async def _loop(self) -> None:
        """Periodic collection loop."""
        while self._running:
            try:
                await self.collect()
            except Exception as e:
                logger.error(f"Collector {self.__class__.__name__} failed: {e}")
            await asyncio.sleep(self._interval)

    @abstractmethod
    async def collect(self) -> None:
        """Perform one collection cycle."""
        ...
```

```python
class SchedulerCollector(BaseCollector):
    """
    Collects scheduler, adapter, and SLA metrics.

    Interval: 30 seconds (configurable).

    Sources:
    - SchedulerHealthAggregator.check() → health status, active adapters, dead streams
    - ConcurrencyManager → active_count, active_by_cadence
    - Redis hydra:stream_failures:* → consecutive failure counts
    - Redis hydra:sla_miss:* → SLA miss events
    - Redis hydra:last_fetch:* → last fetch timestamps for staleness
    """

    def __init__(
        self,
        health_aggregator: SchedulerHealthAggregator,
        concurrency_manager: ConcurrencyManager,
        redis: Redis,
        registry: StreamRegistry,
        interval: float = 30.0,
    ):
        ...

    async def collect(self) -> None:
        """
        One collection cycle:
        1. Call health_aggregator.check() → update hydra_scheduler_health_status.
        2. Read active_count → update hydra_scheduler_active_adapters.
        3. Read active_by_cadence for each cadence → update hydra_scheduler_active_by_cadence.
        4. Scan hydra:stream_failures:* → update hydra_adapter_consecutive_failures per stream.
        5. Count streams exceeding dead_stream_threshold → update hydra_adapter_dead_streams.
        6. Scan hydra:sla_miss:* for new events since last collection → increment hydra_scheduler_sla_misses_total.
        7. For each stream in adapter_health → update hydra_adapter_health_status.
        """
        ...
```

## 7. Alert Rules

Location: `prometheus/rules/hydra_alerts.yml`

### 7.1 Critical Alerts (page immediately)

```yaml
groups:
  - name: hydra_critical
    rules:
      - alert: HydraSchedulerUnreachable
        expr: hydra_scheduler_health_status == 0
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "HYDRA scheduler is UNREACHABLE"
          description: "PostgreSQL or Redis is down. No ingestion possible."

      - alert: HydraStoragePrimaryDown
        expr: hydra_storage_health_status{engine=~"postgres|redis"} == 0
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "Primary storage engine {{ $labels.engine }} is UNREACHABLE"
          description: "System of record unavailable. All writes will fail."

      - alert: HydraBackpressureBlocked
        expr: hydra_backpressure_state == 2
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "Engine {{ $labels.engine }} backpressure BLOCKED for 5m"
          description: "WAQ depth {{ $value }} exceeds hard limit. Tasks being rejected."

      - alert: HydraDLQCritical
        expr: hydra_storage_dlq_depth > 500
        for: 10m
        labels:
          severity: critical
        annotations:
          summary: "DLQ depth for {{ $labels.engine }} exceeds 500"
          description: "Significant data loss risk. Reconciliation failing."

      - alert: HydraAPIDown
        expr: up{job="hydra-api"} == 0
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "HYDRA API is unreachable"
          description: "Prometheus cannot scrape /metrics. API may be down."
```

## 9. Anomaly Detection

Location: `src/hydra/monitoring/anomaly.py`

### 9.1 AnomalyDetector

```python
class AnomalyDetector:
    """
    Statistical anomaly detection on correlation and intelligence metrics.

    Two detection methods:
    1. Z-score: deviation from rolling mean (window: 24 hours of 5-minute samples = 288 points)
    2. EWMA: exponentially weighted moving average with configurable span

    Runs as a background task on configurable interval (default: 300 seconds).
    """

    def __init__(
        self,
        pg_pool: asyncpg.Pool,
        zscore_threshold: float = 3.0,
        ewma_span: int = 24,
        window_size: int = 288,
        interval: float = 300.0,
    ):
        self._pg_pool = pg_pool
        self._zscore_threshold = zscore_threshold
        self._ewma_span = ewma_span
        self._window_size = window_size
        self._interval = interval
        self._history: dict[str, deque[float]] = defaultdict(lambda: deque(maxlen=window_size))
```

### 9.2 Detection Methods

**Z-Score Detection:**
```python
async def _check_zscore(self, metric_key: str, current_value: float) -> tuple[float, bool]:
    """
    Compute z-score of current_value against rolling history.

    Returns (zscore, is_anomaly).
    Requires minimum 30 data points before flagging.
    """
    history = self._history[metric_key]
    history.append(current_value)
    if len(history) < 30:
        return 0.0, False
    mean = statistics.mean(history)
    stdev = statistics.stdev(history)
    if stdev == 0:
        return 0.0, False
    zscore = (current_value - mean) / stdev
    return zscore, abs(zscore) > self._zscore_threshold
```

**EWMA Detection:**
```python

        super().__init__(f"Collector '{collector_name}' failed: {message}")

class AnomalyDetectionError(MonitoringError):
    """Raised when anomaly detection computation fails."""
    pass

class CapacityPlanningError(MonitoringError):
    """Raised when capacity planning computation fails."""
    pass

class SLOComputationError(MonitoringError):
    """Raised when SLO/error budget computation fails."""
    pass
```

All collector errors are caught within the `BaseCollector._loop()` and logged — they never propagate to crash the application. Anomaly, capacity, and SLO errors follow the same pattern within their respective background tasks.
