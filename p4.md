## 1. Purpose

Deliver two specialized adapters for structured statistical and institutional data:
- **OData Adapter** — covers OData v4 services exposing entity sets with standardized query syntax. Targets: WHO GHO, Eurostat (OData endpoint), World Bank (OData endpoint), OECD. Tiers 8, 9, 26.
- **SDMX Adapter** — covers Statistical Data and Metadata eXchange services used by national statistical offices and international organizations. Targets: Eurostat, ECB Statistical Data Warehouse, IMF, OECD, UN Data. Tiers 8, 9.

Both subclass `BaseAdapter` from P1 and produce `NormalizedRecord` output.

## 3. OData Adapter — `src/hydra/adapters/odata.py`

### 3.1 Class: `ODataAdapter(BaseAdapter)`

OData (Open Data Protocol) v4 defines a standardized REST interface for querying and manipulating data. Services expose entity sets queryable via URL conventions with system query options (`$filter`, `$select`, `$orderby`, `$top`, `$skip`, `$count`, `$expand`). Responses are JSON-based (`application/json;odata.metadata=minimal`).

### 3.2 OData Service Discovery

Before querying entity sets, the adapter supports optional service document and metadata discovery:

| Endpoint | Path | Returns | Purpose |
|---|---|---|---|
| Service document | `{base_url}/` | JSON listing of entity sets | Enumerate available collections. |
| Metadata document | `{base_url}/$metadata` | CSDL XML (Common Schema Definition Language) | Schema introspection — entity types, properties, navigation properties, enums. |

Discovery is triggered only when the registry entry declares `odata_discover: true`. Results are cached in memory for the adapter instance lifetime. The metadata document is parsed to extract property names and types for downstream validation.

### 3.3 `fetch` Implementation

- Reads `base_url` (e.g., `https://ghoapi.azureedge.net/api`), `entity_set` (e.g., `GHO`), and query options from the stream registry entry.
- Constructs the request URL: `{base_url}/{entity_set}?{query_options}`.
- System query options assembled from registry fields:
  - `odata_filter` → `$filter` — OData filter expression (e.g., `IndicatorCode eq 'WHOSIS_000001'`).
  - `odata_select` → `$select` — comma-separated list of properties to return.
  - `odata_orderby` → `$orderby` — sort expression.
  - `odata_top` → `$top` — page size (default: 1000).
  - `odata_expand` → `$expand` — navigation property expansion.
  - `odata_count` → `$count=true` — request inline count of total results.
- Dynamic filter injection: if the registry declares `odata_dynamic_filter`, the adapter appends a time-based filter using `{last_fetch_time}`. Template format: `{odata_dynamic_filter}` with `{last_fetch_time}` placeholder resolved to ISO 8601 (e.g., `ModifiedDate gt {last_fetch_time}` → `ModifiedDate gt 2026-04-02T00:00:00Z`).
- Request headers:
  - `Accept: application/json;odata.metadata=minimal`
  - `OData-MaxVersion: 4.0`
  - `OData-Version: 4.0`

**Pagination:**
OData v4 uses server-driven pagination via `@odata.nextLink`. The adapter:
1. Fetches the initial page.
2. Checks the response for `@odata.nextLink`.
3. If present, follows the link to fetch the next page.
4. Repeats until no `@odata.nextLink` is present or `max_pages` (from registry, default: 50) is reached.
5. All pages are accumulated into a single `RawPayload` with concatenated JSON content (stored as a JSON array of page responses).

**Alternative pagination — `$skip`-based:**
Some OData services do not return `@odata.nextLink` but support `$skip`/`$top` manual pagination. If the registry declares `odata_pagination: skip_top`, the adapter:
1. Fetches with `$top={page_size}&$skip=0`.
2. If the response contains `@odata.count` or the number of returned records equals `page_size`, increments `$skip` by `page_size` and fetches again.
3. Stops when fewer records than `page_size` are returned or `max_pages` reached.

**Auth dispatch:**
- `none` — WHO GHO, most public OData services.
- `api_key` — injected as header or query param per registry `auth_key_location` and `auth_key_name`.
- `oauth2_client_credentials` — for services requiring OAuth2 client credentials flow (e.g., some OECD endpoints). The adapter:
  1. Reads `client_id`, `client_secret`, and `token_url` from `HydraSettings.credentials[stream_id]`.
  2. POSTs to `token_url` with `grant_type=client_credentials`.
  3. Caches the access token for its `expires_in` duration minus a 60-second buffer.
  4. Injects as `Authorization: Bearer {token}` on all subsequent requests.
  5. On HTTP 401 during a data request, refreshes the token once and retries.

**Error handling:**
- OData services return structured error responses: `{"error": {"code": "...", "message": "..."}}`. The adapter parses these and includes the server error message in raised `FetchError` exceptions.
- HTTP 400 with OData error body → `FetchError` with server message (bad query syntax).
- HTTP 404 → `FetchError` (entity set not found).
- HTTP 429 → `RateLimitError` with `Retry-After` header honored.

### 3.4 `parse` Implementation

- Decodes `RawPayload.content` as UTF-8 JSON.
- Extracts the `value` array from the OData response — this is the standard location for entity collections in OData v4 JSON format.
- If the response is a single entity (no `value` array), wraps it in a list.
- Strips OData metadata annotations from each record: removes keys starting with `@odata.` and `odata.` prefixes. These are protocol-level metadata, not domain data.
- If `$expand` was used, flattens expanded navigation properties into the parent record using a configurable separator. Registry field `expand_flatten_separator` (default: `_`). Example: if `Country` is expanded and contains `Name`, the flattened key becomes `Country_Name`.
- Handles OData-specific type representations:
  - `Edm.DateTimeOffset` → Python `datetime` (ISO 8601 parse).
  - `Edm.Date` → Python `date`.
  - `Edm.Decimal` → Python `Decimal` (preserve precision).
  - `Edm.Int64` as string → Python `int`.
  - `Edm.GeographyPoint` → extract coordinates for downstream geo normalization.
  - `null` → Python `None`.
- Each parsed record is tagged with `odata_entity_set` and `odata_service_url` for provenance.

### 3.5 `validate` Implementation

- Required-field validation per registry `required_fields`.
- If metadata discovery was performed (`odata_discover: true`), validate that all returned properties exist in the CSDL schema. Unknown properties are logged at WARNING but not dropped (services may extend schemas).
- Type validation against CSDL property types (if metadata was discovered): verify that values conform to declared `Edm.*` types. Coercion failures are logged and the record is dropped.
- Null handling: OData allows nullable properties. If the registry declares `non_nullable_fields` (list), records with `None` in those fields are dropped.
- Deduplication within batch: uses composite key from registry `dedup_key_fields` if declared. Falls back to `compute_raw_hash` on the full record JSON.

### 3.6 Stream Registry Entry Schema (OData-specific fields)

| Field | Type | Required | Description |
|---|---|---|---|
| `entity_set` | `str` | yes | OData entity set name to query. |
| `odata_filter` | `str` | no | Static `$filter` expression. |
| `odata_dynamic_filter` | `str` | no | Time-based filter template with `{last_fetch_time}` placeholder. |
| `odata_select` | `str` | no | Comma-separated `$select` property list. |
| `odata_orderby` | `str` | no | `$orderby` expression. |
| `odata_top` | `int` | no | Page size (`$top`). Default: 1000. |
| `odata_expand` | `str` | no | `$expand` expression for navigation properties. |
| `odata_count` | `bool` | no | Whether to request `$count=true`. Default: false. |
| `odata_pagination` | `enum[next_link, skip_top]` | no | Pagination strategy. Default: `next_link`. |
| `odata_discover` | `bool` | no | Whether to fetch service document and `$metadata` for schema validation. Default: false. |
| `expand_flatten_separator` | `str` | no | Separator for flattening expanded navigation properties. Default: `_`. |
| `non_nullable_fields` | `list[str]` | no | Fields that must not be null. Records with null values in these fields are dropped. |
| `dedup_key_fields` | `list[str]` | no | Fields forming the composite deduplication key. |
| `auth_pattern` | `enum[none, api_key, oauth2_client_credentials]` | yes | Authentication strategy. |

## 4. SDMX Adapter — `src/hydra/adapters/sdmx.py`

### 4.1 Class: `SdmxAdapter(BaseAdapter)`

Statistical Data and Metadata eXchange (SDMX) is an ISO standard (ISO 17369:2013) for exchanging statistical data. SDMX REST API v2.1 defines a standardized URL structure for querying data flows, data structure definitions (DSDs), and observations. Major providers: Eurostat, ECB, IMF, OECD, UN Data, BIS, ILO.

The adapter uses the `pandasdmx` library (already declared in P0 `pyproject.toml`) as the primary interface, with raw HTTP fallback for providers with non-standard implementations.

### 4.2 SDMX Concepts

| Concept | Description | Relevance |
|---|---|---|
| **Agency** | Organization maintaining the data (e.g., `ESTAT` for Eurostat, `ECB` for ECB). | Used in URL construction. |
| **Dataflow** | A specific dataset definition (e.g., `nama_10_gdp` for Eurostat GDP). | Primary query target. |
| **Data Structure Definition (DSD)** | Schema defining dimensions, attributes, and measures for a dataflow. | Used for validation and field mapping. |
| **Dimension** | A categorical axis of the data (e.g., `GEO`, `TIME_PERIOD`, `FREQ`). | Query filters and record fields. |
| **Observation** | A single data point at the intersection of all dimensions. | One observation = one record. |
| **Key** | Dot-separated dimension values identifying an observation (e.g., `A.GDP.EU27_2020.CLV10_MEUR`). | Used in URL path for filtering. |
| **Content constraint** | Declares which dimension values are valid for a dataflow. | Used for query validation. |

### 4.3 `fetch` Implementation

**Primary mode — `pandasdmx`:**
- Instantiates a `pandasdmx.Request` object for the agency declared in the registry `sdmx_agency` field. `pandasdmx` maintains a built-in registry of known SDMX providers and their endpoint URLs.
- For agencies not in the `pandasdmx` built-in registry, the adapter registers a custom source using `base_url` from the stream registry.
- Fetches the dataflow's DSD first (if not cached): `request.datastructure(dataflow_id)`. The DSD is cached in memory for the adapter instance lifetime. This provides dimension names, code lists, and the observation structure needed for parsing.
- Constructs the data query:
  - `dataflow_id` from registry.
  - `key` from registry `sdmx_key` — dot-separated dimension filter (e.g., `A.GDP..` where empty positions are wildcards).
  - `params` dict from registry `sdmx_params`:
    - `startPeriod` — start of time range. Supports `{last_fetch_time}` template variable (resolved to SDMX period format, e.g., `2026-Q1` or `2026-04`).
    - `endPeriod` — end of time range.
    - `dimensionAtObservation` — which dimension varies at the observation level (usually `TIME_PERIOD`).
    - `detail` — `full` (default), `dataonly`, `serieskeysonly`, `nodata`.
    - `updatedAfter` — ISO 8601 timestamp for incremental queries. Maps to `{last_fetch_time}`.
  - Executes: `request.data(dataflow_id, key=key, params=params)`.
- Response is an SDMX message object (parsed by `pandasdmx` internally from XML or JSON).

**Fallback mode — raw HTTP:**
- If `pandasdmx` fails (unsupported provider, non-standard response), the adapter falls back to raw HTTP requests.
- Constructs the SDMX REST URL manually: `{base_url}/data/{dataflow_id}/{key}?{params}`.
- Accepts `application/vnd.sdmx.data+json;version=1.0.0` (SDMX-JSON) or `application/vnd.sdmx.genericdata+xml;version=2.1` (SDMX-ML).
- Preference order declared in registry `sdmx_format_preference` (default: `[json, xml]`).
- Raw response stored in `RawPayload` for manual parsing.

**Auth dispatch:**
- `none` — Eurostat, ECB, most public SDMX services.
- `api_key` — some providers require registration and API key.
- `oauth2_client_credentials` — same flow as OData adapter for providers requiring OAuth2.

**Rate limiting:**
- SDMX providers often have strict rate limits. The adapter respects `Retry-After` headers and implements a configurable `sdmx_request_delay_seconds` (from registry, default: 1.0) between consecutive requests to the same provider.

### 4.4 `parse` Implementation

**`pandasdmx` mode:**
- Converts the SDMX message to a `pandas.DataFrame` via `pandasdmx.to_pandas()`. This produces a DataFrame with a MultiIndex (one level per dimension) and observation values as columns.
- Resets the MultiIndex to produce flat columns — each dimension becomes a regular column.
- Iterates rows. Each row becomes a dict with dimension names as keys plus the observation value column(s).
- Attaches dimension code list labels (human-readable names) as additional fields. For example, if dimension `GEO` has value `EU27_2020`, the adapter adds `GEO_label: "European Union - 27 countries (from 2020)"` using the code list from the cached DSD.
- Handles multiple measures: if the dataflow has multiple observation columns (measures), each measure becomes a separate field in the dict.

**Raw HTTP — SDMX-JSON mode:**
- Parses the JSON response per SDMX-JSON specification.
- Navigates: `dataSets[0].series` → for each series key, `observations` → for each observation.
- Resolves dimension values from the `structure.dimensions` section using positional indices.
- Resolves attribute values from `structure.attributes`.
- Each observation becomes one dict with resolved dimension names and values.

**Raw HTTP — SDMX-ML (XML) mode:**
- Parses XML using `xml.etree.ElementTree`.
- Navigates: `DataSet` → `Series` → `Obs` elements.
- Extracts dimension values from `Series` attributes and `Obs` attributes.
- Resolves code list labels from the cached DSD.
- Each `Obs` element becomes one dict.

**Common post-parse step:**
- Each parsed record is tagged with `sdmx_agency`, `sdmx_dataflow`, and `sdmx_key` for provenance.
- Time period normalization: SDMX uses various period formats (`2026`, `2026-Q1`, `2026-M04`, `2026-04-03`). The adapter normalizes all to ISO 8601:
  - Annual (`2026`) → `2026-01-01T00:00:00Z`
  - Quarterly (`2026-Q1`) → `2026-01-01T00:00:00Z`
  - Monthly (`2026-M04` or `2026-04`) → `2026-04-01T00:00:00Z`
  - Daily (`2026-04-03`) → `2026-04-03T00:00:00Z`
  - The original period string is preserved in a `time_period_raw` field.

### 4.5 `validate` Implementation

- Required-field validation per registry `required_fields`.
- DSD-based validation (if DSD was fetched):
  - Verify that all dimension values in each record exist in the corresponding code list. Unknown dimension values are logged at WARNING. Records with unknown values in dimensions declared as `strict_dimensions` (from registry) are dropped.
  - Verify observation values are numeric where the DSD declares a numeric measure. Non-numeric values are logged and the record is dropped.
- Time period validation: verify the normalized timestamp is within a plausible range. Registry may declare `time_period_range` as `[start, end]` (ISO 8601). Records outside this range are dropped.
- Observation value validation: registry may declare `observation_value_range` as `[min, max]`. Values outside this range are dropped and logged (catches data entry errors like GDP in wrong units).
- Deduplication within batch: composite key formed from all dimension values. Duplicate observations (identical across all dimensions) are dropped.

### 4.6 Stream Registry Entry Schema (SDMX-specific fields)

| Field | Type | Required | Description |
|---|---|---|---|
| `sdmx_agency` | `str` | yes | SDMX agency identifier (e.g., `ESTAT`, `ECB`, `IMF`). |
| `dataflow_id` | `str` | yes | SDMX dataflow identifier (e.g., `nama_10_gdp`). |
| `sdmx_key` | `str` | no | Dot-separated dimension filter. Empty positions are wildcards. Default: all wildcards. |
| `sdmx_params` | `dict` | no | Additional SDMX REST query parameters (`startPeriod`, `endPeriod`, `updatedAfter`, `detail`, `dimensionAtObservation`). Supports `{last_fetch_time}` template. |
| `sdmx_format_preference` | `list[str]` | no | Ordered preference for response format: `json`, `xml`. Default: `[json, xml]`. |
| `sdmx_request_delay_seconds` | `float` | no | Minimum delay between consecutive requests to the provider. Default: 1.0. |
| `strict_dimensions` | `list[str]` | no | Dimension names where unknown code list values cause record rejection (not just warning). |
| `time_period_range` | `list[str]` | no | Valid time period range as `[start_iso, end_iso]`. Records outside are dropped. |
| `observation_value_range` | `list[float]` | no | Valid observation value range as `[min, max]`. |
| `include_dimension_labels` | `bool` | no | Whether to include human-readable code list labels alongside dimension codes. Default: true. |
| `dedup_key_fields` | `list[str]` | no | Override composite dedup key. Default: all dimension fields. |
| `auth_pattern` | `enum[none, api_key, oauth2_client_credentials]` | yes | Authentication strategy. |
