## 1. Purpose

Build the REST API layer that exposes all upstream capabilities to external consumers. P11 owns:

1. **Intelligence product endpoints** — CRUD and on-demand generation for SITREP, Entity Dossier, and Threat Assessment products.
2. **Analytical query endpoints** — record queries, time-series, full-text search across storage engines.
3. **Graph query endpoints** — entity network analysis, shortest paths, centrality ranking.
4. **Correlation endpoints** — on-demand correlation runs, result listing and retrieval.
5. **Timeline endpoint** — cross-tier temporal event sequencing.
6. **Watchlist management** — CRUD for entity and region watchlists.
7. **Health and operational endpoints** — scheduler health, backpressure state, dead streams.
8. **Registry metadata** — tier listing and detail.
9. **Cross-cutting concerns** — authentication, pagination, rate limiting, error handling, response envelope.

After P11, HYDRA is externally accessible — analysts and downstream systems consume intelligence products and run ad-hoc queries via a documented REST API.

## 2. Design Decisions

Architectural choices resolved from the P0–P10 pattern inventory:

### 2.1 Watchlist Persistence — PostgreSQL Table

Config-file reload (`HydraSettings`) does not survive without restart and cannot support concurrent writes. Watchlists get dedicated PG tables with CRUD. `AnalysisSettings` retains default watchlist values for initial seeding only.

### 2.2 Authentication — API Key Header (MVP), JWT-Ready

MVP: `X-API-Key` header validated against a `api_keys` PG table (key hash, name, created_at, expires_at, scopes). No OAuth2 flow for MVP — adds complexity without MVP value. Architecture is JWT-ready: the `get_current_user` dependency is a single swap point for JWT validation post-MVP.

### 2.3 Pagination — Cursor-Based

Cursor-based pagination using opaque `next_cursor` tokens (base64-encoded `(sort_field, last_value, last_id)` tuples). Offset/limit breaks on large, frequently-updated tables (`normalized_records`, `correlation_results`). All list endpoints return `PagedResponse[T]`.

### 2.4 Response Envelope — Standardized Wrapper

All responses use a consistent envelope:

```python
class APIResponse(BaseModel, Generic[T]):
    data: T
    meta: ResponseMeta | None = None
    errors: list[APIError] | None = None

class ResponseMeta(BaseModel):
    request_id: str          # UUID4 per request
    timestamp: str           # ISO 8601 UTC
    duration_ms: float       # server-side processing time
    pagination: PaginationMeta | None = None

class PaginationMeta(BaseModel):
    next_cursor: str | None
    has_more: bool
    total_estimate: int | None  # approximate count, not guaranteed exact

class APIError(BaseModel):
    code: str                # machine-readable error code
    message: str             # human-readable description
    detail: dict[str, Any] | None = None
```

### 2.5 Long-Running Operations — 202 Accepted + Polling

Product generation and correlation runs can take significant time. These endpoints return `202 Accepted` with a `job_id` and a status polling endpoint. No WebSocket/SSE for MVP — polling is simpler and sufficient.

```python
class JobStatus(BaseModel):
    job_id: str              # UUID4
    status: str              # "pending" | "running" | "completed" | "failed"
    progress: float | None   # 0.0–1.0 if available
    result_id: str | None    # product_id or correlation_run_id when completed
    error: str | None        # error message if failed
    created_at: str
    updated_at: str
```

Job state stored in Redis with TTL (default: 1 hour). Completed jobs include `result_id` pointing to the persisted product or correlation run.

### 2.6 Rate Limiting — Redis-Backed, Per-Endpoint

Token bucket algorithm via Redis. Three tiers:

| Tier | Endpoints | Rate | Burst |
|---|---|---|---|
| `read` | GET endpoints (products, records, health, registry) | 100 req/min | 20 |
| `search` | Search, graph, timeline, correlation listing | 30 req/min | 10 |
| `write` | POST endpoints (generate, correlate, watchlist mutations) | 10 req/min | 5 |

Rate limit headers on every response: `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`.

### 2.7 OpenAPI — Auto-Generated from Pydantic Models

FastAPI's native OpenAPI generation from Pydantic v2 models. No hand-curated overrides for MVP. Tags group endpoints by router. Description and examples populated via `Field(description=..., examples=[...])` on schema fields.

    query: Select,
    params: PaginationParams,
    sort_column: Column,
    id_column: Column,
    session: AsyncSession,
) -> tuple[list[Row], PaginationMeta]:
    """
    Apply cursor-based pagination to a SQLAlchemy select.

    1. If cursor provided: decode and add WHERE clause
       (sort_column, id_column) > (last_value, last_id).
    2. Apply ORDER BY sort_column ASC, id_column ASC.
    3. Apply LIMIT (params.limit + 1) to detect has_more.

    8. Return (rows, PaginationMeta).

## 17. Open for P12

P11 exposes the following for P12 (Monitoring & Alerting) consumption:

1. **Health endpoints** — `/api/v1/health`, `/api/v1/health/backpressure`, `/api/v1/health/streams/dead` — P12 polls these for alerting thresholds.
2. **Product metrics** — `intelligence_products` PG table: generation frequency, confidence distributions, completeness trends.
3. **Correlation metrics** — `correlation_results` PG table: correlation volume, confidence distributions, pipeline performance.
4. **Job failure tracking** — Redis job state: failed job counts, timeout rates.
5. **Rate limit metrics** — Redis rate limit keys: consumption patterns, 429 rates.
6. **SLA miss events** — Redis `hydra:sla_miss:*` keys from P8 maintenance DAG.
7. **DLQ depth** — Dead letter queue metrics from P8.
8. **TaskResult history** — Execution metrics via XCom/Redis from P8.

P12 can consume these via:
- Direct API polling (authenticated)
- Direct PG/Redis queries (for metrics aggregation)
- Prometheus exposition endpoint (P12 scope — not P11)
