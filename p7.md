## 1. Purpose

Build the abstraction layer between adapter output (`list[NormalizedRecord]` from `BaseAdapter.run()`) and the six storage engines defined in P0. P7 owns:

1. **Routing** — analyzing each `NormalizedRecord` for data characteristics and dispatching to the correct engine(s).
2. **Deduplication** — Redis-backed `raw_hash` checking at the router level before any engine write.
3. **Serialization** — engine-specific transformation of `NormalizedRecord` into the format each engine accepts.
4. **Write path** — async write-ahead buffer through Redis, drained by per-engine writer tasks.
5. **Connection pooling** — per-engine singleton pools shared across all adapter instances.
6. **Health checks** — per-engine health probes reporting status to the scheduler and API.
7. **Reconciliation** — dead letter queue processing for failed writes with retry and audit trail.

After P7, the adapter pipeline is complete end-to-end: `fetch → parse → validate → normalize → route → store`.

## 3. Architecture

### 3.1 Data Flow

```
Adapter.run() → list[NormalizedRecord]
       ↓
StorageRouter.route(records)
       ↓
  ┌─ Dedup check (Redis hash set) ─── duplicate? → drop, log DEBUG
  │
  ├─ Classify traits per record
  │   ├─ Always: PostgreSQL (primary system of record)
  │   ├─ has geo? → PostGIS spatial index on PG row
  │   ├─ time-series tier? → InfluxDB (secondary)
  │   ├─ text-heavy / large payload? → Elasticsearch (secondary)
  │   ├─ graph-schema tier? → Neo4j (secondary)
  │   └─ binary artifact? → MinIO (blob) + PG (object reference)
  │
  └─ Enqueue to Redis write-ahead buffer (per-engine queues)
       ↓
Engine Writers (async consumer tasks)
  ├─ PostgresWriter    ← drains redis queue "hydra:waq:postgres"
  ├─ InfluxWriter      ← drains redis queue "hydra:waq:influxdb"
  ├─ ElasticsearchWriter ← drains redis queue "hydra:waq:elasticsearch"
  ├─ Neo4jWriter       ← drains redis queue "hydra:waq:neo4j"
  └─ MinioWriter       ← drains redis queue "hydra:waq:minio"
       ↓
  Success → ack, remove from queue, update PG storage_status
  Failure → retry (3x exponential backoff) → DLQ → reconciliation worker
```

## 4. Routing Strategy

### 4.1 Trait-Based Routing with Registry Override

Every `NormalizedRecord` is routed to one or more engines based on data characteristics. The `stream_registry.yaml` gains a new optional `storage` section per tier that can explicitly declare target engines, overriding trait inference.

### 4.2 Default Trait Routing Rules

**Rule 1 — PostgreSQL (always):**
Every `NormalizedRecord` gets a row in PostgreSQL. This is the system of record, the queryable backbone, the thing Alembic migrations manage. Records with `geo` populated get PostGIS spatial indexing automatically via a `geometry` column derived from the GeoJSON.

**Rule 2 — InfluxDB (time-series tiers):**
Records from tiers with `sub_minute` or `realtime` cadence get a secondary write to InfluxDB. Default qualifying tiers:
- Tier 1 (Geophysical & Seismic) — `sub_minute`
- Tier 3 (Space Weather & Solar) — `sub_minute`
- Tier 18 (Aviation & Maritime) — `sub_minute`
- Tier 23 (Space Situational Awareness) — `realtime`
- Tier 24 (Geoscience & Earth Systems) — `realtime`

The InfluxDB write extracts `timestamp`, `stream_id`, selected numeric `payload` fields (declared in registry `influx_fields`), and `tags` as InfluxDB point tags.

**Rule 3 — Elasticsearch (text-heavy tiers):**
Records from text-heavy tiers get a secondary write for full-text search. Default qualifying tiers:
- Tier 14 (Arms & Defense Trade)
- Tier 15 (Conflict & Event Data)
- Tier 16 (Cyber Threat Intel)
- Tier 17 (Social Media & Web OSINT)
- Tier 19 (Sanctions & Financial Intel)
- Tier 20 (NBC Threat)
- Tier 21 (Human Rights)

Additionally, any record where `len(json.dumps(record.payload)) > 4096` bytes qualifies regardless of tier — large payloads benefit from inverted indexing.

**Rule 4 — Neo4j (graph-schema tiers):**
Records from relationship-dense tiers with declared graph schemas get a secondary write as nodes with edges. Default qualifying tiers:
- Tier 16 (Cyber Threat Intel) — ATT&CK kill chains, threat actor → malware → indicator relationships from STIX `relationship` objects.
- Tier 19 (Sanctions & Financial Intel) — entity networks, ownership chains, sanctions list cross-references.
- Tier 15 (Conflict & Event Data) — actor → event → location graphs.

Neo4j writes require a `graph_schema` declaration in the registry (see §4.4). Tiers without a declared graph schema do not get Neo4j writes even if listed above.

**Rule 5 — MinIO (binary artifacts):**
Records where the adapter produced a binary artifact get the blob stored in MinIO. The `NormalizedRecord.payload` stores a MinIO object reference (`bucket` + `key`); the raw binary goes to MinIO. Qualifying conditions:
- FDSN `dataselect` streams (miniSEED waveforms) — Tiers 1, 24.
- S3/Bulk adapter streams with binary file types (`.fits`, `.nc`, `.hdf5`, `.tif`, `.mseed`, `.warc`) — Tiers 3, 4, 17, 22, 24.
- Any record where `RawPayload.content_type` indicates a binary MIME type (`application/octet-stream`, `application/fits`, `application/vnd.fdsn.mseed`, `image/tiff`, `application/x-netcdf`, `application/x-hdf5`).

Detection: the adapter sets a `_binary_artifact` key in `NormalizedRecord.payload` containing `{"content": <bytes>, "content_type": <str>, "original_key": <str>}`. The router detects this key, dispatches the binary to MinIO, and replaces the `_binary_artifact` value with the MinIO reference `{"bucket": <str>, "key": <str>, "size": <int>, "content_type": <str>}` before the PostgreSQL write.

**Rule 6 — Redis (not a persistence target):**
Redis serves as the dedup cache and write-ahead buffer. No `NormalizedRecord` is permanently stored in Redis.

### 4.3 Registry Override Schema

New optional fields in `stream_registry.yaml` per tier:

| Field | Type | Required | Description |
|---|---|---|---|
| `storage_engines` | `list[str]` | no | Explicit list of target engines. Overrides trait inference. Values: `postgres`, `influxdb`, `elasticsearch`, `neo4j`, `minio`. `postgres` is always included even if omitted. |
| `influx_fields` | `list[str]` | no | Payload field names to write as InfluxDB point fields. Required if tier routes to InfluxDB. |
| `influx_tag_fields` | `list[str]` | no | Payload field names to write as InfluxDB point tags. Default: `stream_id`, `tier`. |
| `es_index_prefix` | `str` | no | Elasticsearch index name prefix. Default: `hydra-{tier_id}`. |
| `es_text_fields` | `list[str]` | no | Payload fields to map as ES `text` type (analyzed). Default: all string fields. |
| `es_keyword_fields` | `list[str]` | no | Payload fields to map as ES `keyword` type (not analyzed). |
| `graph_schema` | `dict` | no | Neo4j graph schema declaration. Required for Neo4j routing. See §4.4. |
| `minio_bucket` | `str` | no | MinIO bucket name for binary artifacts. Default: `hydra-{tier_id}`. |

### 4.4 Neo4j Graph Schema Declaration

The `graph_schema` registry field defines how `NormalizedRecord` payloads map to Neo4j nodes and edges.

```yaml
# Example: Tier 16 — Cyber Threat Intel
graph_schema:
  node_label_field: "type"           # payload field whose value becomes the node label
  node_id_field: "id"                # payload field used as unique node identifier
  node_properties:                   # payload fields stored as node properties
    - name
    - description
    - created
    - modified
    - mitre_attack_id
    - cve_id
  edge_rules:                        # rules for creating edges from record data
    - type: "relationship"           # match records where payload.type == "relationship"
      source_field: "source_ref"     # payload field containing source node ID
      target_field: "target_ref"     # payload field containing target node ID
      edge_label_field: "relationship_type"  # payload field whose value becomes edge label
      edge_properties:               # payload fields stored as edge properties
        - description
    - type: "sighting"
      source_field: "sighting_of_ref"
      target_field: null             # sightings are node-only (no target edge)
      edge_label_field: null
```

```yaml
# Example: Tier 19 — Sanctions & Financial Intel
graph_schema:
  node_label_field: "entity_type"
  node_id_field: "entity_id"
  node_properties:
    - name
    - aliases
    - country
    - program
    - list_date
  edge_rules:
    - type: "ownership"
      source_field: "owner_id"
      target_field: "owned_id"
      edge_label_field: null         # static label: "OWNS"
      edge_label_static: "OWNS"
      edge_properties:
        - ownership_percentage
        - effective_date
    - type: "association"
      source_field: "entity_id"
      target_field: "associated_entity_id"
      edge_label_static: "ASSOCIATED_WITH"
      edge_properties:
        - association_type
```

## 5. Deduplication

### 5.1 Redis-Backed Dedup Cache

Before dispatching to any engine, the router checks `raw_hash` against a Redis set.

**Key structure:** `hydra:dedup:{tier_id}` — one set per tier for independent TTL management.

**Algorithm:**
1. For each record in the batch, execute `SISMEMBER hydra:dedup:{tier_id} {raw_hash}`.
2. If member exists → record is a duplicate. Drop it. Log at DEBUG: `{stream_id, raw_hash, action: "dedup_drop"}`.
3. If not a member → add via `SADD hydra:dedup:{tier_id} {raw_hash}` with TTL refresh.
4. Proceed to trait classification and dispatch.

**Batch optimization:** Use `SMISMEMBER` (Redis 6.2+) for batch membership checks — single round-trip for the entire record batch.

**TTL per tier:**

| Cadence | TTL | Rationale |
|---|---|---|
| `sub_minute` | 24 hours | High-frequency streams; older hashes unlikely to recur. |
| `realtime` | 24 hours | Same as sub_minute. |
| `15min` | 48 hours | Moderate frequency. |
| `hourly` | 72 hours | |
| `daily` | 7 days | |
| `weekly` | 30 days | |
| `monthly` | 90 days | |
| `quarterly` | 180 days | |
| `annual` | 365 days | Low-frequency; hashes must persist across long intervals. |

TTL is applied per-key via `EXPIRE` on the set. Individual members inherit the set TTL. The router refreshes the set TTL on every write to prevent expiry during active ingestion.

### 5.2 PostgreSQL Safety Net

PostgreSQL has a unique constraint on `raw_hash` as a secondary dedup mechanism. If Redis loses state (restart, eviction, memory pressure), PG catches the duplicate at insert time.

The `PostgresEngine.store()` method handles `asyncpg.UniqueViolationError` on the `raw_hash` constraint gracefully:
1. Log at DEBUG: `{stream_id, raw_hash, action: "pg_dedup_catch"}`.
2. Do not raise — the record is simply not inserted.
3. Increment a `pg_dedup_fallback_count` metric for monitoring Redis reliability.

## 6. Write-Ahead Buffer

### 6.1 Redis Queue Structure

Each engine has a dedicated Redis list used as a FIFO queue:

| Queue Key | Engine | Consumer |
|---|---|---|
| `hydra:waq:postgres` | PostgreSQL | `PostgresWriter` |
| `hydra:waq:influxdb` | InfluxDB | `InfluxWriter` |
| `hydra:waq:elasticsearch` | Elasticsearch | `ElasticsearchWriter` |
| `hydra:waq:neo4j` | Neo4j | `Neo4jWriter` |
| `hydra:waq:minio` | MinIO | `MinioWriter` |

### 6.2 Queue Entry Format

Each queue entry is a JSON-serialized dict:

```python
{
    "record_hash": str,          # raw_hash for tracking
    "stream_id": str,            # for logging and metrics
    "tier": int,                 # tier ID
    "enqueued_at": str,          # ISO 8601 UTC timestamp
    "attempt": int,              # current attempt number (starts at 1)
    "payload": str,              # JSON-serialized NormalizedRecord
}
```

### 6.3 Enqueue Semantics

`StorageRouter.route()` is synchronous from the adapter's perspective:
1. Dedup check (Redis).
2. Trait classification.
3. For each target engine, `RPUSH` the serialized entry to the engine's queue.
4. Return immediately. The adapter does not wait for engine writes.

This decouples adapter throughput from storage latency entirely.

### 6.4 Dead Letter Queue

Records that fail engine writes after exhausting retries go to a DLQ:

**Key:** `hydra:dlq:{engine_name}`

**Entry format:**
```python
{
    "record_hash": str,
    "stream_id": str,
    "tier": int,
    "target_engine": str,
    "error": str,                # last error message
    "error_type": str,           # exception class name
    "attempt_count": int,        # total attempts made
    "first_failed_at": str,      # ISO 8601
    "last_failed_at": str,       # ISO 8601
    "payload": str,              # JSON-serialized NormalizedRecord
}
```

## 7. Storage Engines

### 7.1 Abstract Interface: `StorageEngine`

Location: `src/hydra/storage/engines/base.py`

```python
class StorageEngine(ABC):
    """Abstract base for all storage engine implementations."""

    @abstractmethod
    async def store(self, records: list[NormalizedRecord]) -> StoreResult:
        """Write a batch of records to the engine.

        Returns StoreResult with counts of successful, failed, and deduplicated writes.
        Raises StorageEngineError on unrecoverable failures.
        """
        ...

    @abstractmethod
    async def health_check(self) -> StorageHealth:
        """Probe engine connectivity and responsiveness."""
        ...

    @abstractmethod
    async def connect(self) -> None:
        """Initialize connection pool. Called once at startup."""
        ...

    @abstractmethod
    async def disconnect(self) -> None:
        """Gracefully close connection pool. Called at shutdown."""
        ...
```

**Dataclass: `StoreResult`**

| Field | Type | Description |
|---|---|---|
| `engine` | `str` | Engine name (`postgres`, `influxdb`, etc.). |
| `stored` | `int` | Records successfully written. |
| `failed` | `int` | Records that failed to write. |
| `deduplicated` | `int` | Records dropped by engine-level dedup (PG unique constraint). |
| `duration_ms` | `float` | Total write duration. |
| `errors` | `list[dict]` | Per-record error details for failed writes: `{record_hash, error}`. |

### 7.2 PostgresEngine

Location: `src/hydra/storage/engines/postgres.py`

**The system of record.** Every `NormalizedRecord` gets a row here.

**Connection pool:** `asyncpg.create_pool()` with:
- `min_size`: 5 (from `HydraSettings.database.pg_pool_min`, default 5)
- `max_size`: 20 (from `HydraSettings.database.pg_pool_max`, default 20)
- `command_timeout`: 30 seconds
- DSN from `HydraSettings.database.postgres_dsn`

**Table schema** (managed by Alembic migration):

```sql
CREATE TABLE IF NOT EXISTS normalized_records (
    id              BIGSERIAL PRIMARY KEY,
    stream_id       TEXT NOT NULL,
    tier            SMALLINT NOT NULL,
    timestamp       TIMESTAMPTZ NOT NULL,
    geo             GEOMETRY(Geometry, 4326),    -- PostGIS, nullable
    payload         JSONB NOT NULL,
    source_name     TEXT NOT NULL,
    source_url      TEXT,
    adapter_type    TEXT NOT NULL,
    access_level    TEXT DEFAULT 'green',
    raw_hash        TEXT NOT NULL UNIQUE,         -- dedup safety net
    ingested_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    confidence      DOUBLE PRECISION DEFAULT 1.0,
    tags            TEXT[] DEFAULT '{}',
    storage_status  TEXT DEFAULT 'pending',       -- pending | complete | partial | failed
    storage_engines TEXT[] DEFAULT '{}',           -- engines this record was dispatched to

    -- Indexes
    CONSTRAINT chk_confidence CHECK (confidence >= 0.0 AND confidence <= 1.0),
    CONSTRAINT chk_tier CHECK (tier >= 1 AND tier <= 28)
);

CREATE INDEX idx_nr_stream_id ON normalized_records (stream_id);
CREATE INDEX idx_nr_tier ON normalized_records (tier);
CREATE INDEX idx_nr_timestamp ON normalized_records (timestamp);
CREATE INDEX idx_nr_raw_hash ON normalized_records (raw_hash);
CREATE INDEX idx_nr_tags ON normalized_records USING GIN (tags);
CREATE INDEX idx_nr_payload ON normalized_records USING GIN (payload jsonb_path_ops);
CREATE INDEX idx_nr_geo ON normalized_records USING GIST (geo) WHERE geo IS NOT NULL;
CREATE INDEX idx_nr_storage_status ON normalized_records (storage_status) WHERE storage_status != 'complete';
CREATE INDEX idx_nr_ingested_at ON normalized_records (ingested_at);
```

**`store()` implementation:**
1. Serialize each `NormalizedRecord` to a row tuple.
2. Convert `geo` (GeoJSON dict) to PostGIS geometry via `ST_GeomFromGeoJSON()`.
3. Extract `source_meta` fields into flat columns.
4. Batch insert using `asyncpg.Connection.copy_records_to_table()` for bulk performance. Fallback to individual `INSERT ... ON CONFLICT (raw_hash) DO NOTHING` for small batches (< 10 records) or when `copy_records` fails.
5. On `UniqueViolationError`: count as deduplicated, not failed.
6. After successful insert, set `storage_status = 'pending'` (updated to `complete` by the reconciliation worker after all secondary engines confirm).
7. Populate `storage_engines` array with the list of engines this record was dispatched to.

**`health_check()` implementation:**
- Execute `SELECT 1` on the pool.
- Measure round-trip latency.
- Check PostGIS availability: `SELECT PostGIS_Version()`.
- Return `StorageHealth` with status `OK` if both succeed, `DEGRADED` if PostGIS fails but PG responds, `UNREACHABLE` if connection fails.

### 7.3 InfluxEngine

Location: `src/hydra/storage/engines/influxdb.py`

**Time-series secondary store** for high-frequency tiers.

**Client:** `influxdb_client.InfluxDBClientAsync` with:
- `url`: from `HydraSettings.database.influxdb_url` (default `http://influxdb:8086`)
- `token`: from credential store, key `influxdb_admin`
- `org`: from `HydraSettings.database.influxdb_org` (default `hydra`)
- `timeout`: 30_000 ms

**Bucket:** `hydra-timeseries` (created on first connect if absent).

**`store()` implementation:**
1. For each `NormalizedRecord`, construct an InfluxDB `Point`:
   - **measurement**: `stream_id`
   - **time**: `record.timestamp`
   - **tags**: `{"tier": str(record.tier), "stream_id": record.stream_id}` + any fields listed in registry `influx_tag_fields`
   - **fields**: numeric values from `record.payload` matching registry `influx_fields`. Non-numeric fields are skipped with a DEBUG log.
2. Batch write using `write_api.write()` with `write_precision=WritePrecision.MS`.
3. On write failure: individual record retry not supported by InfluxDB batch API. The entire batch is retried. If batch fails 3 times, all records in the batch go to DLQ.

**`health_check()` implementation:**
- Call `client.ping()`.
- Measure latency.
- Return `StorageHealth`.

### 7.4 ElasticsearchEngine

Location: `src/hydra/storage/engines/elasticsearch.py`

**Full-text search secondary store** for text-heavy tiers and large payloads.

**Client:** `elasticsearch.AsyncElasticsearch` with:
- `hosts`: from `HydraSettings.database.elasticsearch_url` (default `http://elasticsearch:9200`)
- `max_retries`: 3
- `retry_on_timeout`: True
- `request_timeout`: 30

**Index naming:** `hydra-tier-{tier_id}-{YYYY.MM}` — monthly rotation for manageable index sizes. Alias `hydra-tier-{tier_id}` points to the current month's index.

**Index mapping template:**

```json
{
  "mappings": {
    "properties": {
      "stream_id":    {"type": "keyword"},
      "tier":         {"type": "integer"},
      "timestamp":    {"type": "date"},
      "geo":          {"type": "geo_shape"},
      "payload":      {"type": "object", "dynamic": true},
      "source_name":  {"type": "keyword"},
      "source_url":   {"type": "keyword"},
      "raw_hash":     {"type": "keyword"},
      "confidence":   {"type": "float"},
      "tags":         {"type": "keyword"},
      "ingested_at":  {"type": "date"}
    },
    "dynamic_templates": [
      {
        "strings_as_text": {
          "match_mapping_type": "string",
          "mapping": {
            "type": "text",
            "fields": {
              "keyword": {"type": "keyword", "ignore_above": 256}
            }
          }
        }
      }
    ]
  },
  "settings": {
    "number_of_shards": 1,
    "number_of_replicas": 0,
    "refresh_interval": "5s"
  }
}
```

Registry `es_text_fields` and `es_keyword_fields` override the dynamic template for specific payload fields.

**`store()` implementation:**
1. Serialize each `NormalizedRecord` to an ES document dict.
2. Convert `geo` GeoJSON to ES `geo_shape` format (identical structure — GeoJSON is natively supported).
3. Flatten `source_meta` into top-level fields.
4. Bulk index using `helpers.async_bulk()` with `_index` set to the current month's index.
5. `_id` set to `raw_hash` — natural dedup at the ES level (duplicate `_id` overwrites, which is idempotent).
6. Parse bulk response for per-item errors. Failed items go to DLQ.

**`health_check()` implementation:**
- Call `client.cluster.health()`.
- Status mapping: `green` → `OK`, `yellow` → `DEGRADED`, `red` → `UNREACHABLE`.
- Measure latency.

### 7.5 Neo4jEngine

Location: `src/hydra/storage/engines/neo4j.py`

**Graph secondary store** for relationship-dense tiers.

**Driver:** `neo4j.AsyncGraphDatabase.driver()` with:
- `uri`: from `HydraSettings.database.neo4j_uri` (default `bolt://neo4j:7687`)
- `auth`: from credential store, key `neo4j_admin` → `(username, password)`
- `max_connection_pool_size`: 10

**`store()` implementation:**

The Neo4j write is the most complex because it must interpret the `graph_schema` declaration to create nodes and edges.

1. Load `graph_schema` from the stream registry for the record's tier.
2. If no `graph_schema` exists for this tier, skip (this should not happen — the router only dispatches to Neo4j for tiers with declared schemas, but defensive check).
3. For each `NormalizedRecord`:
   a. **Node creation:**
      - Label: `record.payload[graph_schema.node_label_field]` (e.g., `"attack-pattern"` → node label `AttackPattern`, converted to PascalCase).
      - Unique ID: `record.payload[graph_schema.node_id_field]`.
      - Properties: extract fields listed in `graph_schema.node_properties` from `record.payload`.
      - Add metadata properties: `_stream_id`, `_tier`, `_timestamp`, `_raw_hash`, `_ingested_at`.
      - Cypher: `MERGE (n:{Label} {id: $id}) SET n += $properties`
   b. **Edge creation** (if record matches an `edge_rule`):
      - Check if `record.payload.get("type")` matches any `edge_rules[].type`.
      - If matched and both `source_field` and `target_field` are present in payload:
        - Edge label: from `edge_label_field` in payload, or `edge_label_static` from schema. Converted to UPPER_SNAKE_CASE.
        - Cypher: `MATCH (a {id: $source_id}), (b {id: $target_id}) MERGE (a)-[r:{EDGE_LABEL}]->(b) SET r += $properties`
      - If only `source_field` is present (e.g., sightings): create node only, no edge.
4. Execute all Cypher statements in a single transaction per batch.
5. On transaction failure: retry entire transaction up to 3 times with exponential backoff (1s, 2s, 4s). If all retries fail, all records in the batch go to DLQ.
6. **Label sanitization:** Node labels derived from payload values are sanitized — spaces replaced with underscores, special characters stripped, converted to PascalCase. Edge labels converted to UPPER_SNAKE_CASE. Invalid labels (empty string, numeric-only) cause the individual record to fail with a logged error; the rest of the batch proceeds.
7. **MERGE semantics:** All node and edge creation uses `MERGE` (not `CREATE`) to ensure idempotency. Re-ingesting the same record updates properties without creating duplicates.

**`health_check()` implementation:**
- Execute `RETURN 1` via the driver.
- Measure round-trip latency.
- Return `StorageHealth`.

### 7.6 MinioEngine

Location: `src/hydra/storage/engines/minio.py`

**Binary blob store** for large/binary artifacts.

**Client:** `boto3.client('s3')` with:
- `endpoint_url`: from `HydraSettings.database.minio_url` (default `http://minio:9000`)
- `aws_access_key_id`: from credential store, key `minio_admin`
- `aws_secret_access_key`: from credential store, key `minio_admin`
- `region_name`: `us-east-1` (MinIO default)

**Bucket naming:** `hydra-tier-{tier_id}` — one bucket per tier. Created on first connect if absent.

**`store()` implementation:**
1. For each `NormalizedRecord` with a `_binary_artifact` key in `payload`:
   a. Extract `content` (bytes), `content_type` (str), `original_key` (str).
   b. Construct object key: `{stream_id}/{YYYY}/{MM}/{DD}/{raw_hash}_{original_key}` — partitioned by date for S3 listing performance.
   c. Upload via `put_object()` with:
      - `ContentType`: from `content_type`
      - `Metadata`: `{"stream_id": stream_id, "tier": str(tier), "raw_hash": raw_hash, "ingested_at": iso_timestamp}`
   d. On success, return the MinIO reference dict: `{"bucket": bucket_name, "key": object_key, "size": len(content), "content_type": content_type}`.
   e. The router replaces `_binary_artifact` in the record's payload with this reference dict **before** the PostgreSQL write, so PG stores the reference, not the blob.
2. On upload failure: retry 3 times with exponential backoff. On exhaustion → DLQ.
3. **Size guard:** If `len(content) > 5 GB`, reject with `StorageEngineError` — MinIO single-part upload limit. Multipart upload is deferred to post-MVP.

**`health_check()` implementation:**
- Call `list_buckets()`.
- Measure latency.
- Return `StorageHealth`.

## 8. Async Writer

Location: `src/hydra/storage/writer.py`

### 8.1 AsyncWriter

Each engine has a dedicated `AsyncWriter` instance that consumes from its Redis write-ahead queue.

```python
class AsyncWriter:
    """Per-engine queue consumer with retry and DLQ semantics."""

    def __init__(
        self,
        engine: StorageEngine,
        redis_cache: RedisCache,
        queue_key: str,
        dlq_key: str,
        batch_size: int = 100,
        poll_interval: float = 0.5,
        max_retries: int = 3,
        backoff_base: float = 1.0,
        backoff_factor: float = 2.0,
        backoff_ceiling: float = 30.0,
    ):
        ...

    async def run(self) -> None:
        """Main consumer loop. Runs indefinitely until cancelled."""
        ...

    async def _process_batch(self, entries: list[dict]) -> None:
        """Deserialize, call engine.store(), handle results."""
        ...

    async def _retry_or_dlq(self, entry: dict, error: Exception) -> None:
        """Increment attempt, re-enqueue or move to DLQ."""
        ...
```

### 8.2 Consumer Loop

1. `BLPOP` from the engine's queue with `timeout=poll_interval`. This blocks efficiently without busy-waiting.
2. Accumulate entries up to `batch_size` or until the queue is empty (non-blocking `LPOP` drain after the first `BLPOP` returns).
3. Deserialize entries, reconstruct `NormalizedRecord` instances.
4. Call `engine.store(records)`.
5. For each record in `StoreResult.errors`: call `_retry_or_dlq()`.
6. For successful records: update PostgreSQL `storage_status` via a lightweight query:
   ```sql
   UPDATE normalized_records
   SET storage_status = CASE
       WHEN array_length(storage_engines, 1) = (
           SELECT COUNT(*) FROM unnest(storage_engines) AS e
           WHERE EXISTS (/* check all engines confirmed */)
       ) THEN 'complete'
       ELSE storage_status
   END
   WHERE raw_hash = ANY($1::text[]);
   ```
   This is batched — one query per writer batch, not per record.

### 8.3 Retry Logic

- `attempt` field in the queue entry tracks retry count.
- On failure: if `attempt < max_retries`, increment `attempt`, compute delay = `min(backoff_base * backoff_factor ^ attempt, backoff_ceiling)`, sleep, then `RPUSH` back to the queue.
- On exhaustion (`attempt >= max_retries`): `RPUSH` to the DLQ key with full error context.

### 8.4 Reconciliation Worker

A separate async task that processes DLQ entries on a configurable interval (default: 5 minutes).

```python
class ReconciliationWorker:
    """Processes dead letter queue entries with extended retry."""

    def __init__(
        self,
        engines: dict[str, StorageEngine],
        redis_cache: RedisCache,
        interval: float = 300.0,       # 5 minutes
        max_dlq_retries: int = 5,
        alert_threshold: int = 100,     # DLQ depth triggering alert
    ):
        ...

    async def run(self) -> None:
        """Periodic DLQ drain loop."""
        ...
```

**Reconciliation logic:**
1. For each engine DLQ, `LRANGE` to inspect entries without removing.
2. Group entries by engine.
3. Attempt re-store via `engine.store()`.
4. On success: `LREM` from DLQ, update PG `storage_status`.
5. On failure: increment `attempt_count` in the DLQ entry, update `last_failed_at`.
6. If `attempt_count > max_dlq_retries`: log at ERROR with full context, set PG `storage_status = 'failed'` for the record. The record remains in DLQ for manual inspection.
7. If DLQ depth exceeds `alert_threshold`: log at CRITICAL. (P12 wires this to alerting infrastructure.)

## 15. Startup and Shutdown

### 15.1 Startup Sequence

The storage layer initializes in order:

1. **RedisCache.connect()** — dedup and queue infrastructure must be available first.
2. **PostgresEngine.connect()** — primary store, run Alembic migration check.
3. **InfluxEngine.connect()** — create bucket if absent.
4. **ElasticsearchEngine.connect()** — create index template if absent.
5. **Neo4jEngine.connect()** — verify connectivity.
6. **MinioEngine.connect()** — create buckets if absent.
7. **Start AsyncWriter tasks** — one per engine, as `asyncio.Task`.
8. **Start ReconciliationWorker** — as `asyncio.Task`.
9. **Run StorageHealthAggregator.check_all()** — log initial health status.

If PostgreSQL or Redis fail to connect at startup → **fatal error**, process exits. Secondary engine failures at startup → log WARNING, mark engine as `UNREACHABLE`, continue. Records routed to unreachable engines go directly to DLQ.

### 15.2 Shutdown Sequence

1. **Cancel AsyncWriter tasks** — allow in-flight batches to complete (5-second grace period).
2. **Cancel ReconciliationWorker**.
3. **Disconnect engines** in reverse order: MinIO → Neo4j → Elasticsearch → InfluxDB → PostgreSQL.
4. **RedisCache.disconnect()** — last, since writers may flush final entries during grace period.

## 20. Design Rationale

**Why PostgreSQL as universal primary?**
Every record needs a queryable, ACID-compliant home. PostgreSQL with PostGIS handles structured data, geospatial queries, and JSONB payload search. It's the system of record that P9 (Correlation) and P10 (Analysis) query against. Secondary engines are optimizations for specific access patterns, not replacements.

**Why async write-ahead instead of synchronous?**
Adapter throughput is the bottleneck we're optimizing for. A seismic adapter fetching real-time data cannot block on an Elasticsearch bulk index. The Redis buffer decouples these concerns. The cost is eventual consistency on secondary engines — acceptable for an intelligence platform where the primary store (PG) is always synchronous from the adapter's perspective (via the queue enqueue, which is sub-millisecond).

**Why Redis for dedup instead of PostgreSQL?**
Speed. Redis `SISMEMBER` is O(1) with sub-millisecond latency. PostgreSQL `SELECT EXISTS` on an indexed column is fast but involves connection checkout, query parsing, and disk I/O. At high-frequency ingestion (sub_minute tiers producing thousands of records per cycle), the Redis path is materially faster. PG remains the safety net.

**Why no distributed transactions?**
Complexity cost exceeds benefit at MVP scale. Two-phase commit across PostgreSQL, Elasticsearch, and Neo4j would require a transaction coordinator, increase latency, and introduce failure modes that are harder to debug than a DLQ reconciliation worker. The partial-write-plus-reconciliation approach gives us eventual consistency with full auditability and simpler failure recovery.

**Why per-tier dedup TTL?**
Memory management. A sub_minute tier producing 1,440 records/day doesn't need hashes retained for 30 days. An annual tier producing 1 record/year needs its hash retained for at least a year to prevent re-ingestion. TTL-per-cadence balances memory usage against dedup effectiveness.
